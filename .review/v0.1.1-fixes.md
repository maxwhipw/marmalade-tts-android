# v0.1.1 fixes review

**Verdict:** PASS with notes

## Summary

The four substantive fixes (655c3e6, cabfae0, f795e08, 0394d9c) all do
what their commit messages claim. The launch-crash fix is the right
shape: `viewModel()` at the two out-of-NavHost call sites resolves
against the Activity's HiltViewModelFactory without dragging in
SavedStateHandle, and the four screens inside the NavHost still use
`hiltViewModel()` — verified by grep, no stray call sites. The
seed-race fix is architecturally sound on two levels (combine() in
SpeakViewModel resolves once the catalog flow emits, AND seeding moved
to `MarmaladeTtsApplication.onCreate` so the seed is already running
before any VM is built); three regression tests pin the new invariants
and all 117 unit tests pass. The engine warm-up fix correctly uses
`onLoadLanguage` as the trigger, runs on a serviceScope that's
cancelled in `onDestroy`, and the `synchronized(loadLock)` inside
`ensureModelLoaded()` means a racing `onSynthesizeText` cannot
double-load. Docs reconciliation matches code on disk for SPEC.md,
README.md, CHANGELOG.md, CONTRIBUTING.md, STUBS.md, and
`MarmaladeDb.kt` KDoc.

The "with notes" carve-out is for two non-blocking documentation
misses: (1) `ROADMAP.md` was not touched by the reconciliation, and it
still carries the stale "Oboe + audiofx" claim (lines 25-26, 57) and
still has the v0.1.0 MVP checkbox list as all-unchecked even though
v0.1.0 has shipped; (2) the new STUBS.md entry for `NativeEngineHandle`
references `AppModule.kt:124-126` but the post-cabfae0 AppModule is
shorter and `provideNativeEngineHandle` is now at lines 100-103. Both
are cosmetic but worth fixing before v0.1.2.

## Per-fix verification

### Launch crash (commit 655c3e6)
- **Status:** FIXED
- **Notes:** Replaces `hiltViewModel()` with `viewModel()` at
  `AppRoot.kt:69` and `OnboardingScreen.kt:83` — the two call sites
  outside a NavBackStackEntry. The mechanism described in the commit
  message is correct: `viewModel()` resolves
  `LocalViewModelStoreOwner` to the Activity and uses
  `defaultViewModelProviderFactory`, which on `@AndroidEntryPoint
  MainActivity` is a HiltViewModelFactory that DOES populate
  CreationExtras with SAVED_STATE_REGISTRY_OWNER_KEY (because
  ComponentActivity implements SavedStateRegistryOwner natively).
  `hiltViewModel()` builds its own CreationExtras and omits that key,
  which is the IllegalArgumentException seen in the v0.1.0 stack
  trace.
- **NavHost screens unaffected:** verified all four still use
  `hiltViewModel()` — `SpeakScreen.kt:122`, `EnginesScreen.kt:76`,
  `AliasScreen.kt:89`, `VoicePickerScreen.kt:77`. They live inside
  `composable(...) { }` and get a NavBackStackEntry as their
  ViewModelStoreOwner, which is also a SavedStateRegistryOwner — the
  saved-state path resolves correctly.
- **Grep audit:** no other `hiltViewModel()` call sites in the
  codebase (only the comment in `MainActivity.kt:13` mentions it; that
  KDoc is now slightly stale but harmless — AppRoot's `viewModel()`
  still routes through the same factory).
- **Test coverage:** No new test. Reasonable — testing this would
  require a Compose UI test against a real ComposeTestRule with a
  NavController. The crash is observable on device launch, and the
  commit notes "Verified on a connected Pixel" so this is the right
  spot to lean on manual verification rather than write a brittle
  unit test.

### Seed race + sticky state + scope leak (commit cabfae0)

#### Seed race (Blocker #2)
- **Status:** FIXED
- **Notes:** `SpeakViewModel.currentVoice` now uses
  `combine(settings.defaultVoiceId.onEach {...}, voiceDao.getAll()) {
  id, voices -> voices.firstOrNull { it.id == id } }`. When the
  application-scope seed coroutine populates `voice_meta`, Room's Flow
  for `getAll()` emits the new list and the combine re-evaluates,
  producing the resolved VoiceMeta. The alias-tracking side effect
  stays on the `defaultVoiceId` branch (inside the `.onEach`) so it
  only fires on id changes, not catalog refreshes — correct behaviour
  preserved.
- **One behavioural change worth flagging:** `SharingStarted` changed
  from `Eagerly` to `WhileSubscribed(5_000)`. Side effect: the
  alias-clearing `onEach` only runs while currentVoice has a
  subscriber + 5s grace. Walked through every alias-clearing scenario
  (Speak screen stays mounted; user nav to VoicePicker; user nav to
  Engines for >5s and returns) — DataStore replays current value on
  re-subscription so the side effect fires correctly in every path.
  Test `applyAlias_thenManualVoiceChange_clearsEffect` still passes.
- **Test coverage:** `currentVoice_emitsOnceVoicesArrive` is a real
  regression test — uses a new `MutableVoiceDao` that exposes a
  mutable `Flow<List<VoiceMeta>>`, asserts null while empty, flips to
  populated, asserts non-null resolves to the expected voice. This
  is the right shape for the test.

#### Sticky ModelMissing (Major #1)
- **Status:** FIXED
- **Notes:** `onTextChanged` now resets `_playbackState` to `Idle`
  when current state is `ModelMissing` or `Error`. KDoc explicitly
  notes that `Speaking` is NOT reset because synthesis is still in
  flight — that's the right carve-out.
- **Test coverage:** `onTextChanged_resetsStickyModelMissing` and
  `onTextChanged_resetsStickyError` both exist and pass. Good
  regression coverage.

#### Seed-scope leak (Major #4)
- **Status:** FIXED
- **Notes:** The `RoomDatabase.Callback` and its captured
  `seedScope = CoroutineScope(...)` are gone from `AppModule`.
  Seeding moved to `MarmaladeTtsApplication.onCreate` on an
  application-scoped `CoroutineScope`. Gated by `count() == 0` so
  idempotent across cold starts. The Application is one-per-process
  so the scope is genuinely process-lived without leaking anything
  shorter. Comments in both files explain the why.
- **Test coverage:** No direct test for the leak — appropriate, since
  the bug was latent. `ApplicationTest` still passes (it doesn't
  instantiate). The seed-race regression test indirectly exercises
  the flow that would now consume the seeded data.

### Engine warm-up (commit f795e08)
- **Status:** FIXED
- **Notes:** `onLoadLanguage` override delegates to
  `onIsLanguageAvailable` for the LANG_* result, and on any positive
  match fires `warmUpEngine()` which launches on `serviceScope` (new,
  `SupervisorJob() + Dispatchers.IO`) and calls
  `engine.ensureModelLoaded()` inside try/catch. `serviceScope.cancel()`
  in `onDestroy()`. The race with synthesize is handled correctly by
  the existing `synchronized(loadLock)` + double-check pattern in
  `KittenEngine.ensureModelLoaded()` — if synth fires while warm-up
  is in flight, synth blocks on the same monitor until load completes
  (no double-load, no torn state).
- **Onboarding warm-up not affected:** the `KittenEngine` singleton is
  shared between the UI path (Synthesizer) and the system TTS service
  path; the warm-up on the system TTS service path benefits both.
  Good.
- **Test coverage:** None added. Acceptable — the existing
  `MarmaladeTtsServiceTest` doesn't call `onLoadLanguage`, and adding
  one would require extending `FakeKittenEngine` to record
  `ensureModelLoaded()` calls (currently it doesn't override that
  method — `ensureModelLoaded()` is final on KittenEngine, so the
  base implementation runs and would fail with "isInstalled()=true
  but file paths don't exist" against Robolectric). The fix is small
  enough and the failure mode is observable enough that the deferred
  test is justified.

## Recommendation

v0.1.1 is a clean release as it stands. All four substantive fixes
land correctly, the test suite expands meaningfully (114 → 117), the
APK builds, and the bugs the whole-project review flagged as
ship-blocking (launch crash, seed race) are demonstrably gone — the
seed-race regression test would catch any future regression of the
same shape.

The cosmetic doc drift in ROADMAP.md, STUBS.md line numbers, and
MainActivity.kt's KDoc comment do NOT warrant a v0.1.2 by themselves
— they're cleanup-on-next-touch, not user-visible. If v0.1.2 happens
for other reasons (e.g. one of the deferred Minors), bundle these
fixes in. If the next change is a feature commit, fold the doc
cleanup into that PR.
